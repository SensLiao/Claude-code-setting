# AI-Native Interface — AI 原生交互范式层（架构 / 编排，不是视觉皮肤）

> **干什么**：当一个产品的"前门"不再是导航菜单、而是"开始一场对话 → AI 抛出几个具体选项 → 一来一回收窄"时，本 skill 给出这种产品的**形状决策**和**一场会话的编排规则**。它回答的是"这个产品是什么 SHAPE、一场 session 的 choreography 是什么"——**不回答**"它长什么样"。
>
> **不干什么**：不挑视觉风格（L3 owns），不写视觉底盘（UI-SPEC → `chassis.yaml` owns），不替代 `ux-principles` 的通用命名法则与事后 audit（只往里加两条 conversation-flow 法则，且那是**另一个 agent** 去 wire 的），不碰 build / eng（URL-state 实现、runtime 组装 → GSD / frontend-design），不碰 AI 面的 security（→ `security-app-llm`）。
>
> **来源**：本 skill distill 自一份第一性原理推演文档（Doc3，2026-06，《从第一性原理出发思考 AI 时代网页交互的新范式》）。源文档是一段 reasoning walk，本 skill 把它转成可执行的 doctrine + decision-gate + build-contract。

## §0 这是什么 / 不是什么（边界声明 — 先读这条）

| | |
|---|---|
| **是** | 一层 **paradigm / interaction-architecture**。决定产品形状（是不是"对话即产品"）+ 一场会话的编排（show-then-ask 节奏、骨架库治理、连续界面、首屏唯一任务、开场白姿态、留存铁律）。 |
| **不是** | **不是一种 L3 视觉风格**。它与 L3（taste / luxury / brutalist）**正交**——同一个项目可以"锁定 luxury 作为视觉皮肤" + "本 skill 作为交互范式"，二者**同时成立、互不替代**。 |

**硬声明（本 skill 的存在性约束）**：

- 本 skill **绝不**进任何 `l3_style` enum，**绝不**与任何 L3 风格互斥，**绝不**被 style-lock 成"第 4 种风格"。L3 mutex（taste / luxury / brutalist 一次只挂一个）与本 skill **无关**——本 skill 不参与那个 mutex。
- 本 skill **COMPOSES** on top of 任意一个已锁定的 L3：L3 提供视觉词汇（颜色 / 字体 / 卡片质感 / 间距 token），本 skill 提供"什么时候出哪个解法形态、对话怎么走"。
- 在 routing 谱系里，本 skill 坐在 `ux-principles` **旁边**（都是横切 / foundation 层），不抢 `uiux-product-orchestrator` 的编排权，也不抢任何 L3 skill 的视觉权。

> 一句话记牢：**L3 决定"它好看成什么样"，本 skill 决定"它是不是一场对话、这场对话怎么编排"。两者叠加，不二选一。**

## §1 第一性原理（不能搞反的那条）

源文档的起点只有一句话，是后面一切推理的地基：

> **网页把"你必须先知道才能问"这件事，翻转成了"你看到了，才知道自己想要什么"。**

命令行时代，你必须事先知道要敲什么命令，系统不开口你就什么都得不到。网页把"可能性"（按钮 / 链接 / 卡片）摆出来，让你**看一看、点一点**地意识到自己想要什么。

**AI / 自然语言改变了什么——这条最容易搞反，务必钉死方向**：

LLM 让用户能用日常语言说"我想让生意做大一点"。这看上去像是"既然能直接说，就不必把选项摆出来了"。**错。方向是反的：**

> **自然语言输入让"展示选项"变得 _更_ 必要，而不是 _更不_ 必要。**

原因：用户打出"我想让生意做大一点"时，他**自以为**说清楚了，其实没有——这句话能指十种完全不同的方向，连他自己都还没意识到要哪一种。**他要看到具体的东西，才会知道。** 光靠用户几句话判断不出真实需求，所以"展示选项"反而是刚需。

> ⚠️ 反模式护栏：任何"既然有 AI 了就让用户自己描述、AI 直接照做"的设计，都踩反了第一性原理。AI 越强，越要**先展示具体猜测**让用户挑，而不是把理解负担全压给一问一答。

## §2 Show-Then-Ask 排序规则（决策门）

面对一个模糊的需求，有两条推进路：

- **方式 A**：AI 不停追问，慢慢把意图收窄（理解负担全压在 AI 一侧 + 用户对着白纸答题）。
- **方式 B**：AI 先快速抛出 2–3 个**具体**的猜测，让用户选或修正（看到具体的东西反而帮用户想清楚）。

二者都存在，真正的问题是**谁先来**。答案：

> **先展示，后追问（Show-Then-Ask）。**

**决策门（照此判定，别凭感觉）**：

| 当前状态 | 动作 |
|---|---|
| 用户意图**不清晰**（开局 / 一句模糊诉求 / 刚分享一点信息） | **先展示** —— 抛 2–3 个具体猜测："你是想做这种？这种？还是这种？" |
| 用户意图**已经清晰**（用户已选定某个方向 / 明确否定了某些猜测） | **再追问** —— 这时追问细节才有清晰目标 |

为什么顺序不能反：AI 一上来就问细节，而用户连大方向都没想清楚 → 用户回答越来越含糊 → AI 顺着错误方向越挖越深 → 双方都在浪费时间。先抛 2–3 个具体猜测，用户很快能说"第二个，但某一点要改" → 对话才真正有了起点。

> **命名归属**：把这条 conversation-flow 法则**正式命名并写进 named-laws 正本的，是 `ux-principles`**（由另一个 agent wire 进去）。本 skill 只负责**怎么执行**这条排序门，不持有它的命名权。

## §3 骨架库（核心 artifact + 硬规则）

§2 说"展示选项"，那**选项从哪来？** 不能是 AI 当场凭空编的（显得随意、不可信）。答案是公司维护一个 **骨架库（skeleton library）**。

**骨架 = 一种已验证过的解法形态**——可能是一项服务、一种产品功能、一种应对某类常见问题的典型路径。**注意：骨架不是视觉组件**，它是"解法形态"，不是"卡片样式"。

产品形态因此根本改变：产品不再是事先打包好的固定整体，而变成 **骨架 + 组合规则**，AI 按每个用户的真实情况**实时组装**。

**这里有个陷阱**：如果 AI 完全自由组装，每个用户看到的"产品"都不一样 → ① 用户分不清这是精心匹配还是 AI 瞎拼；② 用户搞不清这家公司到底能做什么、不能做什么。

**硬规则（本 skill 最核心的一条约束）**：

> **AI 只能在公司已验证过的骨架库范围内做组合，绝不自创新骨架。**

这样既让每个用户得到贴近自己的方案，又让公司在用户心里保持清晰、可信的形象。

### §3.1 骨架库 ≠ `chassis.yaml`（两个 artifact 别混）

| Artifact | 是什么 | 归谁 |
|---|---|---|
| **骨架库** `skeleton-library.md` | **解法形态清单**：每条是一种"已验证的回应某类需求的方式" | **本 skill owns** |
| **视觉底盘** `chassis.yaml` | **视觉词汇**：颜色 / 字体 / 卡片质感 / 间距 token / 允许的视觉单元 | **UI-SPEC → `uiux-product-orchestrator` / 锁定的 L3 owns**（本 skill 不写） |

二者正交：骨架库说"此刻该回应哪种解法形态"，chassis 说"用什么视觉单元把它画出来"。本 skill **绝不**把骨架库当视觉 chassis 用（见 §12 反模式）。

### §3.2 `skeleton-library.md` 参考结构（小骨架，不是模板倾倒）

每条骨架建议带这几个字段（落到 `<project>/.planning/` 或项目约定位置，本 skill 只给结构，不替项目写第一版内容）：

```md
## skeleton: <id>            # 稳定标识，如 sk-cashflow-tightening
problem_pattern: <一句话>     # 它回应的需求 / 痛点模式（用户处境长什么样时该抽它）
evidence_tier: team | client # 证据档位（见 §4）：team=团队判断 / client=真实客户证据
                              # client 档需附形如 "50 家里 30 家" 的观察来源
allowed_compositions:         # 允许和哪些骨架组合呈现（防止 AI 乱拼）
  - <other-skeleton-id>
  - ...
notes: <可选>                 # 边界 / 不适用场景 / 已知坑
```

> `id` / `problem_pattern` / `evidence_tier` / `allowed_compositions` 是最小集。AI 在一场对话里"挑 2–3 个最合适的骨架组合呈现"时，**只能从这份清单里挑，且只能按 `allowed_compositions` 组合**。

## §4 骨架库成熟度模型（两个生命阶段）

骨架库本身从哪来？它有两个生命阶段——`evidence_tier` 字段就编码这两档：

| 阶段 | 数据状态 | AI 的姿态 | 推荐依据 |
|---|---|---|---|
| **阶段一** | 还没积累足够客户数据 | "我们**猜**你大概是这种情况，对吗？" | **团队的判断**（`evidence_tier: team`） |
| **阶段二** | 已服务过足够多客户 | "在 50 家和你类似的公司里，有 30 家都遇到过这个具体问题，并且是用这种方式解决的。" | **真实证据**（`evidence_tier: client`，须附观察来源） |

**真正的回报不只是"骨架库更准"**：更重要的是整场对话的气质从 **"我们觉得"** 变成 **"我们见过"**。这种**身份转变**，比单纯"数据更精准"本身更能让客户信任你。

> 设计含义：阶段一不要假装有客户证据（别编"50 家里 30 家"）——诚实标 `team` 档。随客户数据积累，把对应骨架升到 `client` 档并附真实观察来源。这条诚实性是产品可信度的来源，不能为了"显得权威"而造假证据档位。

## §5 设计语言 = 固定词汇（视觉细节 delegate 出去）

如果选项是 AI 实时生成的，那**什么是固定不变的**？答案：**设计语言**——AI 被允许使用的那套**视觉词汇**：有哪几种卡片可出现 / 对比信息怎么排 / 按钮和表单长什么样 / 颜色·间距·字体的统一规则。

AI 的工作是决定"此刻用哪几个视觉单元、用几个、怎么排、填什么内容"，**不能创造新的视觉单元**，只能在已有范围内重组。

这和语言本身同理：几千个常用汉字 + 一套语法 → 写出无穷无尽的句子。**对元素的限制不限制创造力，恰恰是因为有共享结构，创造力才能发挥。**

> **归属（本 skill 不越界）**：这套"视觉词汇"的具体内容——**token / 组件 / 卡片质感 / 颜色阶梯 / 间距**——由**锁定的 L3 skill（taste / luxury / brutalist）+ `design-systems:` 系列** 持有。本 skill **不重复** taste / luxury 的 token 系统，只声明"必须存在一套有限视觉词汇、AI 只能在其内重组"这条**架构约束**。具体 9-shade palette、字号阶梯、双层 bezel 这些细节请去对应 L3 skill 查。

## §6 没有页面——连续界面（continuous interface）

传统网站里，页面是最基本单位：点链接 → 跳新页面，页面像楼里的房间、导航菜单是平面图。但如果 AI 能按当下需要实时生成内容，这整套结构就多余了。

> **不再有彼此分隔的"页面"。整个产品是一个不断更新的连续界面——你看到的画面就是这场对话此刻状态的视觉呈现。**

三个根本后果：

1. **"跳到一个页面"消失了**：浏览器**后退键 = 在对话里退回一步**，不是"回到上一个页面"。
2. **URL 意义变了**：URL 指向的是**一场对话被保存下来的某个状态**（详见 §7）。
3. **传统内容组织方式失效**：不再用主题 / 栏目 / 层级让用户自己导航——**AI 在合适的时候主动把合适的内容推送出来**，用户不再自己找路。

**新问题：用户容易迷失方向**。人的记忆依赖空间感（"那东西是在前一页看到的"）。全变成一片流动画面，用户会找不到自己刚才在哪。

**方向感解药（必须保留 — 可见的对话轨迹）**：保留一些**对话过程中的可见标记**，让用户有"我知道自己走到哪了"的踏实感，又不必把"页面"请回来。Doc3 给的候选形态：

- 侧边栏列出对话走过的**路径**；
- 把过去出现过的卡片以**堆叠**形式保留供翻看；
- **面包屑**式路径。

> 具体选哪种视觉形态是 Doc3 的 open question（见 §13），本 skill 只硬性要求"必须有可见对话轨迹"，不锁死它长什么样（视觉实现归 L3 + build）。

## §7 URL = 对话状态（continuous-interface 的副产物）

承 §6 第 2 条：在连续界面里，

> **URL 存的是会话位置，把 URL 发给别人 = 把这场对话进行到的位置分享给对方。**

这是产品语义层的决策（"URL 代表什么"归本 skill 声明）。但——

> **技术实现是开放问题，交给 build / eng**：把对话状态序列化进 URL、恢复、分享的具体技术方案，**走 GSD / frontend-design**（URL-state、runtime 组装是 eng 层的活）。本 skill **只声明语义**（URL = 对话状态），**不写实现**。Doc3 自己也把"URL-state 技术方案"列为 open question（见 §13）。

## §8 首屏 = 开始对话（start-the-conversation）

传统公司官网想在首屏同时说服很多事（我们是谁 / 做什么 / 服务过谁 / 为什么好），于是堆多个并列按钮（产品 / 案例 / 关于 / 联系）。**这不适合新模式**——新模式下真正的产品是"用户与 AI 的那场对话"，所以首屏**只承担一件事**：

> **说服访客开始这场对话。**

首屏只回答**三个问题**，且只回答这三个：

1. **我点下去会发生什么？** —— 让访客明白接下来是一场 AI 对话，不是填表、不是浏览产品目录。
2. **为什么是 AI，而不是普通表单？** —— AI 能做表单做不到的：即时回应、不要钱、想聊多深就多深。
3. **为什么是 _你们家的_ AI，而不是 ChatGPT？** —— 背后有一个持续积累的**同类企业数据集**，必要时还接入真人顾问。**这第三问最难答好、也最关键**——它决定访客愿不愿意点那个按钮。

**两条首屏硬规则**：

- **只放一个主 CTA**（"开始咨询" / "Consult" / 别的），避免多个并列按钮分散注意力。
- **用"进行中的对话片段"预览替代 logo 墙 / 产品截图**：让访客点击前就先看到这场对话大概长什么样——回到第一性原理：**人是看到了才决定，不是听别人说了才决定。**

> **归属（本 skill 不越界）**：首屏的**视觉执行细节**（排版、字号、hero 结构、动效）defer 给 `luxury-editorial-site-builder` / 锁定的 L3 skill。本 skill 只锁"首屏唯一任务 = 开始对话 + 单主 CTA + 三问 + 对话片段预览替 logo 墙"这层**信息架构决策**，不画像素。

## §9 做过功课的开场白（homework-opener）

用户点主按钮进入对话页后，**谁先开口？** 如果 AI 先说"你好，请问有什么可以帮您？"——它就和市面任何聊天机器人一样，公司的独特价值在这一刻完全看不见。更好的做法：

> **用户先给一点点输入（通常是公司网址或公司名），然后 AI 用"已经做过功课"的姿态开口。**

**入口只需要一个输入框**：系统**自动识别**输入的是 URL 还是公司名，据此选 AI 的开场策略——**不必让用户在入口选两个不同按钮**。

**两条分支**：

| 用户输入 | AI 开场姿态 |
|---|---|
| **公司网址** | "我看了你们的网站。你们做的是 [行业]，技术栈用 [工具]，团队规模看起来在 [人数] 左右。在这个阶段，公司常见的一个具体问题是 [某个具体痛点]——你们有没有遇到过？" —— 即**用已观察到的事实 + 一个可能痛点**开口。 |
| **没有公司网站**（很多小企业靠社媒 / 线下运营，本就没独立站） | 换一种姿态：**承认掌握的信息有限** → 先确认聊的是哪一家公司（**避免重名搞错**，消歧）→ 请用户**用自己的话**讲讲他们做什么。 |

带网址那条开场同时做了两件事：① 展示这个 AI 不是通用的——此刻就已知道具体的东西；② 给用户一个**具体的东西去反应**，比对着白纸描述自己的问题容易得多（再次呼应 §1）。

## §10 两条留存铁律（整套做法里最大的风险）

传统网页用户关标签页时没有情绪（没付出什么努力，走就走）。但**对话不一样**：用户一旦开始分享自己公司的信息，就已经投入了情感。如果他中途带着"我的时间被浪费了 / 我刚才好像被审问"的不爽离开 —— 留下的就是对这家公司的**负面印象**。

> **这是整套新做法里最大的风险，必须从一开始就当作设计的前提。**

用户中途生气离开的根因只有两种：① 觉得**付出很多却什么都没得到**（AI 一直问下一个问题，从没给过有用反馈）；② 觉得**自己在被审问**（问题太快、太深、太私人，没有喘息空间）。

对应两条铁律（无需几个月研究就能立住）：

1. **先给予，再索取，并且一直给（give-before-take）**：每收到用户几条信息后，AI **必须主动返还一条具体的、有用的观察**——**不是**"好的明白了 / 感谢分享"这种敷衍，而是**一条用户此前并没想到的真观点**。
2. **让"离开"变得简单而明显（easy-exit）**：对话每一步都有一个清晰可见的按钮，写诸如"**直接给我看报告**"。用户必须始终感到**停下来这件事由我自己说了算**。觉得"被困住"的用户，比觉得"我随时能走"的用户**走得更快**。

> **命名归属**：把这两条 conversation-flow 铁律**正式命名并写进 named-laws 正本的，是 `ux-principles`**（由另一个 agent wire 进去）。本 skill 负责**怎么执行**（返还节奏、退出按钮存在性），不持有命名权。
>
> 更细的设计（多久返还一次、问题怎么措辞、退出按钮放哪）需要 conversation-design 心理学研究（Doc3 open question，§13）——但上面两条原则本身已站得住脚。

## §11 与系统其余部分的组合（越界护栏 — 本 skill 不做什么由这节钉死）

本 skill 是范式层，**很多东西它声明边界但不亲自做**。下表是 handoff 矩阵——每一行都是"这件事归别人，本 skill 不碰"：

| 关注点 | 归谁 | 本 skill 的角色 |
|---|---|---|
| **视觉风格 / 皮肤**（颜色 / 字体 / 卡片质感 / 动效强度） | **L3 skill**（taste / luxury / brutalist，单独 + **同时**锁定，受 L3 mutex 约束） | 只 COMPOSE，不挑风格、不进 mutex |
| **视觉词汇 / token / 组件**（9-shade、字号阶梯、双层 bezel…） | 锁定的 L3 + `design-systems:` | 只声明"必须有有限视觉词汇 + AI 只在其内重组"（§5），不写 token |
| **视觉底盘** `chassis.yaml` | **UI-SPEC → `uiux-product-orchestrator`**（GSD `/gsd-ui-phase`） | 不写 chassis；骨架库 ≠ chassis（§3.1） |
| **通用命名法则 + 事后 audit**（NN 10-heuristic / Laws of UX / BFM teardown） | **`ux-principles`** | 只往里**加两条 conversation-flow 法则**（§2 / §10），且**由另一个 agent wire**；本 skill 不替代 ux-principles |
| **路由进入本 skill** | **`uiux-product-orchestrator`**（通过另一个 agent 新加的入口） | 被路由方，不抢编排权 |
| **build / eng**（URL-state 实现、runtime 组装、首屏像素实现） | **GSD / frontend-design**（`luxury-editorial-site-builder` 做首屏视觉执行） | 只声明语义 / 信息架构（§7 / §8），不写实现 |
| **AI 面的 security**（prompt injection / tool-perm / 数据外泄 / memory poisoning） | **`security-app-llm`**（上游 `appsec-security-orchestrator`） | AI 对话面一旦上 production，security 走那条线，本 skill 不做安全 |

> 这一节本身就是**防止本 skill 越界的护栏**：凡是落在上表"归谁"列里的事，本 skill 一律 handoff，不亲自做。

## §12 反模式（碰到立刻停手）

1. **聊天机器人硬贴在网站上**：传统多页官网 + 角落一个 chat widget —— 这**不是** AI 原生范式（Doc3 结论：这既不是聊天机器人，也不是"传统网站 + 聊天机器人"，是**完全不同形态**的产品）。
2. **AI 自创骨架**：AI 当场凭空编解法形态，不从已验证骨架库里挑 —— 违反 §3 硬规则，砸掉可信度。
3. **先审问后展示（Ask-Then-Show）**：意图还模糊就开始一问一答追细节 —— 把 §2 决策门**做反**，越挖越偏。
4. **把第一性原理搞反**：因为"有 AI 了"就让用户自己描述、AI 直接照做、不再展示具体选项 —— §1 红线。
5. **困住用户**：没有"直接给我看报告"这类随时可退的明显出口 —— 违反 §10 easy-exit，制造负面印象。
6. **只索取不给予**：AI 一路追问，从不主动返还真观察（或只回"收到 / 感谢分享"敷衍）—— 违反 §10 give-before-take。
7. **logo 墙 / 产品截图当首屏**：而不是用"进行中的对话片段"预览 —— 违反 §8。
8. **首屏堆多个并列 CTA**：产品 / 案例 / 关于 / 联系一起上 —— 违反 §8"单主 CTA"。
9. **入口让用户选"网址 / 公司名"两个按钮**：而不是一个输入框自动识别 —— 违反 §9。
10. **把骨架库当视觉 chassis**：往骨架库里塞颜色 / 字体 / 卡片样式 —— 混淆 §3.1 两个 artifact（骨架库 = 解法形态；chassis = 视觉词汇）。
11. **试图把本范式 style-lock 成 L3**：把 `ai-native-interface` 当成"第 4 种视觉风格"挂进 `l3_style` enum、或让它与 taste / luxury 互斥 —— 直接违反 §0 存在性约束。它是**正交层**，**永远与某个 L3 同时成立**。
12. **阶段一造假证据档位**：没有客户数据却编"50 家里 30 家"冒充 `client` 档 —— 违反 §4 诚实性。

## §13 出处与开放问题

**出处**：本 skill distill 自 Doc3 —— 《从第一性原理出发思考 AI 时代网页交互的新范式》（2026-06，一段 reasoning walk 形式的推理笔记）。源文档自己的态度值得继承：

> 这记录的是一段**推理过程**，不是一份完成的设计方案。如果新的事实推翻了其中某一步，那么从那一步往后的所有结论都应被重新审视。

**Doc3 自带的开放问题（继承下来，本 skill 不假装已解决）**：

1. **对话设计的心理学**：节奏控制、信任建立、人在不同情境愿意分享什么、遇到用户反对怎么处理 —— 研究结果能让 §2 / §10 的原则更精细（但不动其方向）。
2. **骨架库的第一版**：在还没有任何客户数据之前，团队应先把哪些解法形态写下来作为初始骨架（`evidence_tier: team` 的冷启动清单）？
3. **对话轨迹的视觉形态**：§6 的"可见对话轨迹"具体长什么样——侧边栏？堆叠卡片？面包屑？还是别的？（视觉实现归 L3 + build。）
4. **URL-state 技术方案**：把对话状态保存 / 恢复 / 分享通过 URL 实现的具体技术方案（归 build / eng，见 §7）。

> 这 4 个 open question **不影响整体方向的确立**——§0–§12 的范式决策可以照用，这些只是待精化的下游细节。
